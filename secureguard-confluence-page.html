01 — Problem statement

Security is a delivery bottleneck

Developers spend significant time identifying, understanding, and remediating security issues — work that slows delivery, reduces productivity, and often gets deferred under deadline pressure. When it gets deferred, vulnerabilities reach production.

⚠️
The Mythos finding (Anthropic, April 2026)

Anthropic's Claude Mythos model can find vulnerabilities in minutes that previously took skilled researchers months. More than 99% of the vulnerabilities it has already discovered remain unpatched. The median time from vulnerability discovery to weaponised exploit has collapsed from 771 days in 2018 to under 4 hours today.

The three failure modes

🔍
Discovery without remediation

Scanners surface findings but developers don't know how to fix them. The finding sits in a backlog. Time to fix: days or weeks.

🚧
Security as a late-stage gate

Security reviews happen at the end of the cycle. Findings block release. Rework is expensive. Developers experience security as friction, not help.

📋
Generic fixes, not contextual ones

Existing tools generate boilerplate advice. A Java developer sees "use PreparedStatement" — but not how to apply it to their specific 40-line DAO class.

🔓
No hard enforcement

Advisory warnings are ignored under deadline pressure. Without a policy gate that actually blocks merge, the tool is optional and inconsistently used.

02 — Why now

Three forces converging in 2026

"Discovery will no longer be the bottleneck. Remediation will."

— Platform Engineering Magazine, April 2026, on the impact of AI-powered vulnerability discovery

Force 1: AI coding assistants increase surface area

GitHub Copilot writes code faster than ever. Every developer in the organisation already uses it. But AI-generated code inherits the same vulnerability patterns as human-written code — and moves faster into PRs. The attack surface is growing at the speed of AI generation.

Force 2: Mythos proved scale changes everything

Traditional vulnerability management cycles span weeks. Anthropic's Mythos model finds vulnerabilities in minutes. The gap between discovery velocity and remediation velocity is now measured in orders of magnitude. Organisations that don't automate remediation will drown in findings they cannot fix fast enough.

Force 3: Developers already have the workflow

Every developer in this organisation has GitHub Copilot in VS Code. The chat panel, the PR integration, the branch protection hooks — all of it already exists. The missing piece is a security agent that plugs into that workflow and turns it into a security enforcement layer. That is what SecureGuard builds.

03 — Our solution

SecureGuard AI — what it does

SecureGuard is an AI security agent that pairs every finding with an AI-generated explanation, a working code patch, and a unit test — delivered directly into the developer's existing workflow. It does not require new tools, new logins, or new dashboards.

✅
Design principle: zero new tools for developers

SecureGuard surfaces everything through GitHub Copilot Chat (@secureguard), inline VS Code squiggles, git pre-commit hooks, and GitHub PR comments — surfaces developers already work in every day.

The four surfaces

1
VS Code background daemon
Scans on every file save. Red squiggles appear within 2 seconds. No LLM call — fast and free.
IDE · advisory only
2
Git pre-commit hook
Scans staged files on git commit. Blocks CRITICAL findings. Runs in 3–8 seconds. Developers can --no-verify to bypass (logged).
Commit · blocks CRITICAL
3
GitHub Actions PR gate
Deep scan on PR open. Posts inline AI fix suggestions. Blocks merge on HIGH/CRITICAL. Cannot be bypassed — only overridden with the security-override label (logged and audited).
PR · blocks HIGH + CRITICAL

A fourth surface — GitHub Copilot Chat (@secureguard scan this file) — gives developers on-demand deep analysis with streamed AI remediations. This is where Claude does the heavy lifting: reading the actual code, understanding the context, and generating a patch that fits the existing codebase.

05 — Architecture

High level design

SecureGuard is a single Express service that all four surfaces call. The router decides whether to invoke Claude (LLM path: Copilot Chat + PR gate) or return scanner results directly (no-LLM path: IDE + pre-commit). This keeps IDE scanning under 2 seconds and eliminates unnecessary API costs.

The critical routing decision

The single most important architectural choice: Claude is only called on /copilot and /pr routes. The IDE daemon and pre-commit hook use Semgrep and Gitleaks directly — no API call, no latency, no cost. This keeps file-save scanning under 2 seconds and commit hooks under 8 seconds. Java's SpotBugs compile step runs only on full-mode scans (PR/Copilot) — never on file save.

06 — Enforcement

Three enforcement layers

Layer Trigger Scope Tools Speed Blocks on Bypassable?
1 — IDE daemon File save Open file only Semgrep fast · Gitleaks 1–2 sec Nothing — advisory N/A
2 — Pre-commit git commit Staged files only Semgrep · Gitleaks · audits 3–8 sec CRITICAL severity Yes — --no-verify (logged)
3 — PR gate PR open / push All changed files · full dep tree Semgrep full · SpotBugs · OWASP DC 30–90 sec HIGH + CRITICAL No — security-override label only (audited)
📌
Running the same check twice is by design — not duplication

Layer 1 runs fast, lightweight rules on every save — running full Semgrep rulesets on save would make VS Code sluggish. Layer 3 runs Semgrep's full 3,000+ rule set, cross-file dataflow analysis, SpotBugs bytecode analysis, and OWASP Dependency-Check — none of which is feasible on a 2-second budget. These are complementary, not redundant.

08 — Low level design

Component internals

The core service is structured as a set of single-responsibility modules, each independently testable. Here is the responsibility and interface of every module.

Scanner module — scanner.ts

Manages all scanner subprocess invocations. Uses child_process.spawn (never exec) with args as arrays to prevent shell injection. Every scanner runs with a timeout enforced via Promise.race.

MethodLanguageModeToolTimeout
runSemgrep()AllFast + FullSemgrep30s
runGitleaks()AllFast + FullGitleaks15s
runBandit()PythonFast + FullBandit20s
runNpmAudit()JS/TS/AngularFast + Fullnpm audit20s
runPipAudit()PythonFast + Fullpip-audit20s
runSpotBugs()JavaFull onlyMaven compile + SpotBugs120s compile + 60s scan
runOwaspDependencyCheck()JavaFull onlyOWASP Dep-Check180s
⚙️
Java fast-mode caveat

SpotBugs requires compiled bytecode. On file save in VS Code, the IDE daemon skips SpotBugs and shows a hint: "Run mvn compile for Java bytecode analysis." Semgrep still runs in fast mode and catches the most common patterns (SQL injection, hardcoded secrets, XXE).

AI remediator — remediator.ts

Builds a language-aware prompt, calls Claude Sonnet 4.6, and parses the structured XML response (<explanation>, <patch>, <test>) into a Remediation object.

Claude prompt structure (language = Java)
You are SecureGuard AI, an expert in Java application security.

VULNERABILITY: SQL Injection (CWE-89) · CRITICAL · A03:2021
FILE: UserController.java, Line 14
LANGUAGE: Java

VULNERABLE CODE:
stmt.executeQuery("SELECT * FROM users WHERE id = " + userId)

COMPANY GUIDANCE:
Use PreparedStatement or JPA named parameters. Never concatenate
user input. Apply least-privilege to DB accounts.

JAVA FIX NOTE:
Include full import statements. Show complete method, not just
the changed line. If Spring Security config changes are needed,
show the SecurityConfig snippet.

Respond with:
<explanation>Plain English — 2-4 sentences</explanation>
<patch>Fixed Java code with imports</patch>
<test>JUnit 5 + Mockito test</test>

The remediator runs up to 5 concurrent Claude API calls per PR scan. Failures are logged at WARN and skipped — partial results are returned rather than failing the entire scan. Streaming is used for Copilot Chat responses; batch for PR comments.

Policy engine — policy.ts

Evaluates Finding[] against configurable rules and calls the GitHub Statuses API to set the PR gate.

DecisionConditionGitHub statusEffect
PASSNo HIGH or CRITICAL findingssuccessMerge button enabled
BLOCKAny HIGH or CRITICAL finding unresolvedfailureMerge button grayed out
OVERRIDEBLOCK conditions met + security-override label presentsuccessMerge allowed; override recorded in audit log with approver name

Audit log — audit.ts

Every scan writes an AuditRecord to SQLite (production: Postgres). The audit module never throws — audit failure must not break the main scan flow. Records are immutable once written.

Key queries the security team can run:

SQLite — most common vulnerabilities by language
SELECT f.language, f.cwe_id, COUNT(*) AS count
FROM findings f
GROUP BY f.language, f.cwe_id
ORDER BY count DESC;
SQLite — override rate by team (repo prefix)
SELECT
  SUBSTR(repo_path, 1, 30) AS team,
  SUM(policy_decision = 'OVERRIDE') * 1.0 / COUNT(*) AS override_rate
FROM audit_records
WHERE created_at > datetime('now', '-30 days')
GROUP BY team
ORDER BY override_rate DESC;
09 — Workflows

Flow 1 — IDE background scan

Triggered on every file save. Completes in under 2 seconds. No developer action required.

1
Developer saves file in VS Code
LSP client fires onDidSave. Extracts file path and language.
2
POST /scan to localhost:3000
source: "ide", scanMode: "fast". Router receives request, detects language.
3
Scanner runs Semgrep fast rules + Gitleaks
Subprocess with timeout 8s. Java: Semgrep only (no SpotBugs compile on save).
timeout: 8s · no LLM called · partial results if timeout
4
Normalizer maps RawFinding[] → Finding[]
Dedup by (filePath + lineStart + cweId). Drop LOW severity in fast mode.
5
LSP formatter returns Diagnostic[] → VS Code squiggles
CRITICAL/HIGH → red underline. MEDIUM → yellow. Problems panel updated. Hover shows CWE + one-line hint.

Flow 2 — Pre-commit hook

Triggered by git commit. Scans staged files only. Fast enough not to feel like a tax on every commit.

1
Developer runs git commit
Husky pre-commit hook fires. Runs secureguard scan --staged --fail-on CRITICAL
2
Scanner runs on staged files
Semgrep fast + Gitleaks + npm/pip audit. Java: Semgrep only. Total: 3–8 seconds.
3
Decision: CRITICAL found?
No CRITICAL → exit 0 → commit proceeds. MEDIUM/LOW → print warning, commit proceeds. CRITICAL → exit 1 → commit blocked.
exit code 1 blocks git commit · --no-verify bypasses (logged in audit)
4
CLI prints colored finding + hint
CWE ID, file, line, one-line guidance from registry. "Run @secureguard explain <id> for AI fix."

Flow 3 — PR gate (GitHub Actions)

Triggered on PR open or new push. Full deep scan. Cannot be bypassed without the override label.

1
PR opened or new commit pushed
GitHub Actions fires on pull_request event. Runner has Semgrep, Gitleaks, SpotBugs, OWASP DC installed.
2
Full deep scan on all changed files
Java repos: Maven compile → SpotBugs → OWASP Dep-Check. All: Semgrep full ruleset + Gitleaks. Timeout: 90s Semgrep, 180s Dep-Check.
3
Claude generates remediations for HIGH/CRITICAL
Up to 5 concurrent API calls. Language-aware prompts. Java: includes import statements. Angular: includes TS + HTML split.
model: claude-sonnet-4-6 · max_tokens: 1500 · stream: false
4
PR review comments posted via GitHub Suggestions API
One comment per finding. Includes explanation, diff suggestion (one-click apply), OWASP reference link.
5
Policy gate sets commit status
HIGH/CRITICAL unresolved → failure status → merge button grayed out. Override label present → success with override note in audit.
context: 'secureguard/policy-gate' · required in branch protection rules
6
Developer applies fix → re-push → gate passes → merge
Each re-push re-triggers the full scan. Audit record written after every run.

Flow 4 — Copilot Chat (@secureguard)

On-demand deep analysis with streamed AI remediations. This is the highest-value surface for developer understanding.

💬
Supported Copilot Chat commands

@secureguard scan this file · @secureguard scan repo · @secureguard what's vulnerable here? · @secureguard fix the XXE in XmlParser.java · @secureguard /explain <findingId>

The Copilot Extension server (running on localhost:3000 or a deployed URL) receives the developer's message, runs the appropriate scanner, calls Claude with the full code context and registry guidance, and streams the response back as GitHub Flavored Markdown — severity badge → plain-English explanation → fixed code (language-correct fences) → unit test → OWASP reference.

For Angular findings that span both TypeScript component and HTML template, the response includes two separate labeled code blocks. For Java, the response always includes the necessary import statements.

11 — Language support

Supported languages

LanguageDetectionSAST ScannerDep scannerIDE speedStatus
Python .py extension Semgrep + Bandit pip-audit 1–2 sec ✓ Ready
TypeScript / JS .ts .tsx .js .jsx Semgrep npm audit 1–2 sec ✓ Ready
Angular *.component.ts or angular.json Semgrep + Angular rules npm audit 1–2 sec ✓ Ready
Java .java or pom.xml Semgrep + SpotBugs* OWASP Dep-Check Semgrep only (no compile) ✓ Ready

* SpotBugs requires compiled bytecode and runs in full mode only (PR gate + Copilot Chat deep scan). Maven compile runs automatically on the GitHub Actions runner before SpotBugs. If compile fails, a JavaBuildError is returned with a clear message — it does not crash the scan.

Adding a new language in the future

The architecture is language-agnostic at the core. Adding a new language requires exactly three things:

  1. Add a scanner method in scanner.ts (e.g. runGosec() for Go)
  2. Add Semgrep YAML rules to semgrep-rules/
  3. Add CWE entries to remediator-registry.json with language-specific guidance

Nothing else changes — the server, router, policy engine, audit log, and all formatters work unchanged.
